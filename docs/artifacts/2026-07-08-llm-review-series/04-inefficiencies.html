1큰 그림: 네 갈래의 낭비

이 시스템은 매일 저녁 ~2,550개 전 종목의 시세를 갱신하고 지표를 계산한 뒤, 소수 후보만 LLM으로 보낸다. 낭비는 주로 "전 종목을 상대하는 앞단"에서 발생한다. 크게 네 갈래다:

~7,600회
하루에 나가는 외부 API 호출 (KRX ~5,100 + DART ~2,500) — 대부분 "변한 게 없다"는 답을 받으러 간다
~21만 행
하루에 다시 쓰는 DB 행 버전 — 실제 신규는 ~2,550행, 나머지 ~99%는 기존 행 재기록(대부분 같은 값으로 추정 — 미측정)
106MB
무한 증가 중인 freeze 보관함 (1,236개) — 청소 코드는 완성돼 있는데 실행 주체가 없었다
연 ~12일
휴장일에도 헛도는 전체 파이프라인 (거래일 게이트 부재)
먼저 알아둘 것 — 이미 수리가 시작된 항목들

이 감사의 상위 발견 중 3건은 이미 오픈 PR로 리뷰 대기 중이다: DART 호출 일괄화(PR #13), freeze 청소 스케줄 편입(PR #11), 테스트 baseline 0화(PR #12). 리포 잔존물의 .gitignore 정리도 PR #4로 나와 있다. 아래에서는 해당 항목에 PR 배지를 달아 표시한다 — 이들의 최우선 액션은 "새 작업"이 아니라 "리뷰하고 머지"다.

2파이프라인 컴퓨트 — 반복 낭비

C1 — 같은 KOSPI 지수를 종목마다 다시 읽는다

확실

일봉 지표 계산은 종목별 루프인데, 모든 종목에 공통인 KOSPI 지수 시계열을 루프 안에서 매번 DB에서 다시 읽는다(indicators/modes.py:141). 하루에 같은 쿼리 ~2,550회, 중복 전송 ~74만 행. 주봉 체인도 같은 패턴이다.

루프 진입 전에 한 번 읽어 인자로 전달. 몇 줄짜리 수정으로 하루 DB 왕복 2,550회가 사라진다.

C2 — 30일치를 매일 4중으로 다시 쓰는 지표 테이블

조건부

일봉 지표 증분은 안전마진으로 최근 30일 창을 다시 계산하는데, 그 창의 ~5.3만 행에 매일 upsert(있으면 갱신·없으면 삽입) + RS UPDATE + 스크리너 UPDATE + 주봉 게이트 UPDATE의 4중 쓰기가 들어간다. 값이 어제와 같아도 무조건 재기록되는 구조라 하루 행 버전 ~21만 개가 생긴다(PostgreSQL은 UPDATE 때 행을 새로 쓰고 옛 행을 남기므로, 같은 값이어도 쓰기 비용이 든다). 그중 어느 비율이 실제 "같은 값 재기록"인지는 미측정이다 — RS 등급처럼 과거 날짜 값이 정당하게 매일 바뀌는 컬럼도 있다. DB는 자동 청소(VACUUM)로 버티지만 I/O는 매일 지불한다.

트레이드오프: 30일 창은 늦은 데이터 정정을 흡수하는 보험이다. 다만 가격 정정은 이미 별도의 drift 경로가 전 기간 재적재로 처리하므로, 이 보험은 상당 부분 중복이다.

창을 5일 안팎으로 줄이거나, UPDATE에 "값이 실제로 달라졌을 때만" 가드(IS DISTINCT FROM)를 넣기. 안전마진 축소는 검토 후 결정.

C3 — 시세 수신: 전 종목 × 2회 호출로 매일 30일치 재수신

조건부

매일 저녁 종목당 원시가+수정가 2회씩 ~5,100번의 KRX 호출로 각 30일치를 다시 받는다(신규는 사실상 1일치). pykrx에는 "날짜 하나로 전 종목 시세"를 받는 API도 있어서, 원시가 경로는 30일×2시장 ≈ 60호출로 대체 가능하다. 수정가는 종목 단위 API뿐이라 부분 개선만 가능.

원시가만 날짜 단위 조회로 전환하는 부분 개선 검토. 수정가 갱신은 drift 경로 의존도를 높이는 설계 검토와 함께.

C4 — 공시 수집: 종목별 2,500회 호출 → 기간 일괄 조회

확실 PR #13 대기

매 평일 아침 활성 종목 각각에 DART 공시를 물어봐서 ~2,500회 호출이 나가는데, 대부분 "조회 결과 없음" 응답이다. DART는 회사를 지정하지 않으면 기간 전체 공시를 한 번에(페이지당 100건) 주므로, 일일 증분은 1~수 회 호출이면 끝난다(5년 백필도 21청크) — 호출 ~99% 절감.

이미 PR #13(기간 일괄 조회 + 역매핑)이 나와 있다. 리뷰 후 머지.

C5 — 토요일 전체 스윕과 자잘한 중복들

조건부 사소
  • 토요일 전체 스윕 — 전 종목 수정가 재확인(sleep만 ~4분, 전체 수십 분). 평일 공시 기반 경로가 같은 기간을 커버하므로, 스윕이 잡는 것은 "공시가 놓친 경우"뿐. 안전망 가치는 있으니 주기 완화(격주/월간)나, 평일 경로가 못 보는 종목만 골라 도는 차집합 스윕으로.
  • sanity 풀스캔 — 적재 후 검증 COUNT 쿼리에 날짜 조건이 없어 매일 ~130만 행 테이블을 통째로 스캔. 이번 적재 범위로 WHERE 한정하면 끝(수정 극소).
  • market_context — 매 저녁 30일×2지수를 date 루프로 재계산(실제 신규 2행). 신규 날짜만 처리로 전환 가능.
  • 평일 드리프트 이중 수신 — 감지 단계에서 받은 수정가를 버리고 적재 단계에서 다시 받는다.
  • 빌더 이중 계산 — LLM payload를 만들 때 minervini 상세·기업행위 블록을 두 번 계산.
  • 끝이 막힌 컬럼 — 일봉 거래대금(value)은 주봉 value 집계에만 소비되는데, 정작 그 주봉 value를 읽는 곳이 없다(지수의 value는 LLM용 CSV로 소비되므로 예외). 사슬의 끝이 막혀 있는 셈. (stocks.listed_at 전체 NULL은 기지·보류.)

3스케줄 구조 — 헛돌거나, 시간 갭에 기대거나

S1 — 휴장일에도 전체 체인이 돈다

확실

daily 체인(cron 평일 18:30)에 "오늘이 거래일인가" 확인이 없다. 설·추석·임시휴장 등 연 ~12일, KRX 호출 5,100번을 포함한 전체 체인(시세→지표→시장맥락→LLM 러너)이 아무 의미 없이 돈다. 거래일 판정 함수는 이미 코드에 있고 주봉 체인에서만 쓰인다.

체인 진입부에 거래일 가드 한 줄 (기존 expected_latest_trading_day 재사용).

S2 — 시간-갭 직렬화와 이중 실행

조건부 사소
  • 고정 시각 의존 — 토요일 데이터 체인(03:00, 전체 스윕 포함 20~40분 추정)과 주말 LLM(03:20) 사이가 고정 20분 갭이다. 데이터가 늦게 끝나면 LLM이 덜 된 주봉을 읽고, 일찍 끝나면 시간을 버린다. 평일 18:30→19:30→20:00도 같은 구조.
  • performance 이중 실행 — 평일 20:00 통합 실행에 이미 포함된 성과 백필이 23:00에 또 돈다(주말 23:00은 신규 가격이 없어 아무 일도 안 하고 끝난다). 실비용은 미미하나 구조 정리 대상.

후속 잡을 시각이 아니라 선행 잡 완료에 체이닝(데이터 체인 꼬리에 편입 or 성공 후 kick). performance는 한 곳만 남기기.

S3 — dry-run 리허설이 freeze까지 저장한다

조건부

LLM 러너는 아직 실전 대기(dry-run) 상태인데, dry-run이어도 차트 렌더 + 인라인 빌드 + freeze ZIP 디스크 저장까지 전부 실행한다. 리허설로서 렌더·빌드는 가치가 있지만, 재현할 분류가 없는 dry-run freeze는 보존 가치가 낮은 순수 낭비다 — 아래 D1의 106MB 중 상당수가 이것이다.

dry-run일 때 freeze 저장 skip (플래그 하나).

4죽은 무게 — 스토리지와 잔존물

D1 — freeze 보관함 106MB, 무한 증가 중

확실 PR #11 대기

data/freezes가 1,236개·106MB로 증가 중(5월 165 → 6월 1,020개). 90일 보존·활성 보호 로직을 갖춘 청소 코드는 완성돼 있는데, 어떤 cron에도 등록돼 있지 않았다. 실행 주체 없는 청소부.

PR #11(스케줄 편입)이 이미 나와 있다. 리뷰 후 머지. S3(dry-run freeze skip)과 함께면 증가 속도 자체도 줄어든다.

D2 — 리포와 스크립트의 잔존물

사소 PR #4 일부
  • 워킹트리 방치물: 백테스트 산출물 디렉토리 6.8MB(평가 번들·리콜 CSV 등), expert-inquiry 자료, 속도 리포트, 주말 분류 CSV — 커밋/이동/삭제 결정 필요 (.gitignore 정리는 PR #4).
  • 일회성 검증 스크립트 6종(특정 시점 리플레이·재측정) — 아카이브 이동 대상.
  • scripts/cron.example이 현행 crontab과 다른 옛 구성을 안내 — 삭제하고 코드가 생성하는 목록을 정본으로.
  • run_tracking 테이블은 보존 정책 없음(하루 ~8행이라 당장 무해).

5사람 손 — 반복 수작업

T1 — "테스트 실패 ~25개는 원래 그래요"를 매번 사람이 센다

확실 PR #12 대기

테스트 스위트에 격리 문제로 인한 baseline 실패가 ~25개 있어서, 모든 작업 세션이 "이번 실패 수가 늘었나?"를 눈으로 판독해왔다. 회귀 신호의 신호 대 잡음비를 갉아먹는 대표적 toil. 원인 진단(스키마 누수 + 죽은 테스트)은 이미 끝나 있었고, PR #12가 baseline을 0으로 만든다(그 과정에서 가짜 초록 2건도 발견됨).

PR #12 리뷰 후 머지 — 머지 후부터는 "실패 = 회귀"라는 단순 규칙이 성립한다.

T2 — 손으로 맞추는 동기화 2종

조건부
  • schema.sql 이중 수동 적용 — 스키마 변경 시 운영·테스트 두 DB에 각각 psql -f를 사람이 실행. 빠뜨리면 INSERT/테스트가 깨진다. 기동 시 버전 체크 또는 make 타겟 한 개로 해소 가능.
  • 임계값 3원 동기화 — 임계 변경 시 UI 생성 스크립트 재실행과 프롬프트 문구 갱신이 수동. 전자는 pre-commit 훅으로 자동화 가능, 후자는 drift 테스트 확장(PR #5)이 감시망을 넓혀준다.

6효율적이라고 확인한 것들

이번에도 "의심했지만 무죄" 목록을 남긴다 — 다음 감사가 같은 곳을 다시 파지 않도록.

  • RS 등급 벌크 UPDATE — 행별 UPDATE가 아니라 임시 테이블 + JOIN 방식, 주석에 성능 근거까지 남아 있다.
  • LLM 러너의 조회 배치화 — 활성 종목 현재가/전일가를 각 1쿼리로. N+1(목록 1번 + 항목마다 1번씩 총 N+1번 조회하는 안티패턴) 없음.
  • 헛 호출 방지 설계의 일관성 — 이어하기, 멱등 가드(같은 작업을 두 번 실행해도 결과가 한 번과 동일하도록 하는 장치), 7일 중복 방지, 사용량 한도 즉시 중단이 전 경로에 있다 (2편 §3과 일치).
  • 부분 인덱스가 핫 쿼리와 정확히 정합 — 후보 선별 쿼리 조건과 인덱스 조건이 일치.
  • pykrx 타임아웃 주입 — 외부 API 무한 대기 사고의 재발 차단.
  • freeze의 단일 빌드 재사용 — LLM 입력과 감사 아티팩트를 한 번의 빌드로 만들어 이중 렌더가 없다.
