이슈 #30 — 분배일 flag 재계산 ops (+ 검토 중 발견된 선행 사안)

PR #28 머지 후속 수동 절차 4건의 추적 이슈(검토 결과 3건 유효·1건은 전제 오류로 해소) · 최신 main(027bf54)·실DB 재검증 · 검토 중 일일 파이프라인 정지(크론 PATH 결함) 신규 발견

0. 이 문서가 다루는 것

핵심 요약 어제 머지한 PR #28은 "분배일 세는 기준"을 고쳤는데, 새 기준은 앞으로 계산되는 날에만 적용됩니다. 과거 데이터는 옛 기준으로 계산된 채 DB에 남아 있어서, 한 번 전체 재계산을 돌려야 수리가 완성됩니다. 이슈 #30은 그 재계산과 딸린 확인 작업 4가지를 잊지 않기 위한 추적표입니다(PR #28이 Closes #20이라 머지 순간 원래 이슈가 자동으로 닫혀서, 남는 수동 절차가 잊히는 것을 막으려고 만든 이슈 — 출처: PR #28 코드리뷰). 그리고 이번 재검토 중 더 급한 문제를 발견했습니다 — 매일 저녁 자동으로 도는 데이터 수집이 7월 7일부터 멈춰 있습니다.
용어뜻
분배일 (distribution day)주가가 0.2% 이상 하락하면서 거래량은 평소(50일 평균)보다 많은 날 = 기관이 판 흔적. 25일 안에 4개 이상 쌓이면 그 종목을 "관찰(watch)"로 낮추는 근거가 됨.
flag (distribution_day_flag)"이날이 분배일이다"라는 표시. 매일 밤 코드가 계산해 DB의 daily_indicators 테이블에 저장. 분석 LLM은 이 표시를 정답으로 사용.
재계산 (full recompute)과거 전체 기간의 지표를 새 코드로 다시 계산해 DB를 덮어쓰는 것. 계산은 결정론적(같은 입력 → 항상 같은 결과)이라 몇 번 돌려도 안전.
B-수치이 프로젝트 규칙 용어 — "임계값을 바꿨으면 바꾼 효과를 수치로 기록해 두고 나중에 재검토하라"는 의무. 어기고 방치했다가 사고 난 전례(P2-1a)가 있어 규칙이 됨.
handle_quality차트 패턴의 "손잡이(handle)" 구간 품질을 코드로 검사하는 결정론 게이트. 손잡이 구간에 분배일이 1개라도 있으면 distribution_in_handle 경고를 냄.
halt (거래정지)거래가 정지된 날. 시가·고가·저가·거래량은 비어 있고(NULL), 종가는 직전 값으로 채워져(carry) 보존됨 — 이 보존 덕분에 해제일의 등락률 계산이 가능(§5의 핵심 사실).

1. 최신 상태 재검증 결과 이슈 유효 + 선행 사안 발견

핵심 요약 이슈의 4개 항목은 오늘 기준 전부 유효합니다(재계산 아직 안 됨 — 옛 기준 과대 집계분 3,038건 그대로). 추가로, 매일 18:30에 도는 데이터 수집 크론이 7월 7일부터 전부 실패 중임을 발견했습니다. 원인은 코드가 아니라 크론 설정 한 줄(PATH) 문제로 확인됐고, 고치기 쉽습니다.

이슈 전제 재확인 (실DB 측정, 2026-07-11)

  • 최근 1년 flag 78,762건 중 옛 기준(0% 컷)으로만 성립하는 과대 집계분 3,038건(3.9%)이 그대로 남아 있음 → 재계산 미실행 확정.
  • PR #28 머지 이후 새 코드로 계산된 날이 아직 하루도 없음(아래 파이프라인 정지 때문) — 즉 현재 DB의 flag는 100% 옛 기준.
신규 발견 — 일일 파이프라인 정지 (이슈 #30보다 선행)

DB의 최신 데이터: 가격·지표 2026-07-06, 시장 맥락(market_context) 2026-06-29. 오늘이 07-11이므로 거래일 4일치(07-07~07-10)가 통째로 비어 있습니다. (시장 맥락은 크론 정지 이전부터 이미 뒤처져 있었을 가능성이 있어, 따라잡기 실행 때 함께 원인 확인이 필요합니다.)

원인(확정): 크론 로그에 uv: command not found 70회 — 크론탭에 PATH 설정이 없어서, 크론이 실행될 때 uv(파이썬 실행 도구, 실제 위치 /opt/homebrew/bin/uv)를 찾지 못해 모든 작업이 시작 즉시 실패합니다. 마지막 성공 로그는 07-06 저녁이고, 07-08 아침에 크론탭을 정리·재설치한 이력이 있어 그때 PATH 줄이 빠진 것으로 추정됩니다(추정 — 재설치 전 크론탭 원본이 없어 단정 불가, 다만 현재 크론탭에 PATH가 없다는 사실 자체는 확정).

영향: 분배일 flag만이 아니라 가격 수집·전체 지표·시장 상태 판정이 모두 멈춘 상태입니다. 이걸 먼저 고치지 않으면 #30의 재계산을 해도 "오늘까지"가 아니라 "7월 6일까지"만 정합해집니다.

2. 항목① — 과거 데이터 재계산

핵심 요약 매일 밤 자동 계산은 최근 30일 창만 다시 계산합니다. 그래서 새 기준은 저절로 과거까지 소급되지 않고, 전체 기간 재계산을 한 번 돌려야 과거 flag가 새 기준으로 통일됩니다. 안 하면 "옛 기준 과거 + 새 기준 최근"이 섞인 데이터를 분석이 정답으로 쓰게 됩니다.

왜 문제인가 — 실제 흐름: 주말 분류 LLM은 종목마다 최근 60일치 지표 표를 받고, 그중 25일 창의 분배일 개수로 "기관 매도 누적" 판정을 합니다(4개 이상이면 강등). 재계산 전에는 이 60일 표의 앞부분이 옛 기준 flag입니다. 예를 들어 진짜 분배일 2개 + 옛 기준에서만 성립하는 가짜 2개 = 4개로 집계되면, 새 기준으로는 강등감이 아닌 종목이 강등됩니다. 60일이 지나면 최근 창은 자연히 새 기준으로 채워지지만, 백테스트나 과거 조회는 영구히 오염된 채 남습니다.

실행 방법 (검증된 진입점):

uv run python -m kr_pipeline.indicators --target=daily --mode=full-refresh   # 필수 — 분배일 flag는 일봉
uv run python -m kr_pipeline.indicators --target=weekly --mode=full-refresh  # 선택 — 이번 변경과 무관, 함께 돌리면 더 오래 걸림

결정론 계산이라 여러 번 돌려도 같은 결과(안전)이고, 실패해도 다시 돌리면 됩니다. 전 종목·전 기간이라 수십 분 단위가 걸릴 수 있습니다(과거 전체 재계산 전례 있음).

3. 항목② — B-수치 기록 (변경 효과 계량)

핵심 요약 기준을 바꿨으면 "얼마나 달라졌는지"를 숫자로 남기는 게 이 프로젝트의 규칙입니다. 머지 전에 미리 재둔 기준선이 있으므로(과대 집계분 3.9%), 재계산 후 같은 측정을 반복해 전/후를 기록하면 됩니다.

기록할 것: ① 재계산 전후 flag 총수(기준선: 최근 1년 78,762건 중 3,038건 감소 예상) ② §6 강등 경계(25일 내 4개 이상)에 걸리는 종목 수의 변화. 이 수치는 나중에 "종목 분배일에도 시장처럼 변동성 보정(σ 보정)을 넣을지" 재검토할 때의 근거 데이터가 됩니다.

4. 항목③ — handle_quality 발동률 확인

핵심 요약 분배일 flag를 쓰는 곳이 LLM 분석 말고 하나 더 있습니다 — 손잡이 품질 검사 코드입니다. 이 검사는 flag 단 1개로 경고가 켜져서 기준 변화에 민감하므로, 재계산 전후 발동률 비교에 포함해야 합니다.

맥락: 이 소비처는 원래 의존성 점검표에서 빠져 있다가 PR #28 코드리뷰에서 발견돼 점검표에 추가된 곳입니다(그래서 이슈에 별도 항목으로 올라감). 손잡이 구간은 수개월을 걸치기 때문에, 재계산 전에는 같은 손잡이가 "옛 기준 행 + 새 기준 행"을 섞어 보게 되어 검사 결과가 시점에 따라 달라질 수 있습니다 — 재계산이 끝나야 이 검사도 일관됩니다.

5. 항목④ — 거래정지 해제일 미탐지? 검토 결과: 전제가 틀림 — 해소

핵심 요약 이슈는 "정지 해제 첫날의 하락은 분배일로 못 잡는다"고 걱정했는데, 이번 검토에서 실데이터로 확인한 결과 정상적으로 잡히고 있었습니다. 걱정의 근거였던 "전일 종가가 비어 있다(NaN)"는 가정이 사실과 달랐기 때문입니다 — 이 시스템은 정지일에도 종가를 직전 값으로 채워서(carry) 보관합니다. 따라서 이 항목은 결정할 것이 없어졌고, 대신 잘못된 가정을 적어 둔 코드 주석 한 곳을 정정해야 합니다.

왜 걱정했었나: PR #28 리뷰에서 "정지일 데이터는 NaN이니, 해제 첫날의 등락률 계산도 NaN이 되어 그날 폭락+대량 거래(전형적 기관 매도)가 flag에서 빠진다"는 지적이 나왔고, 저도 그 가정 위에 코드 주석("재개일 갭다운 분배는 미탐지")을 남겼습니다.

실데이터 확인 (2026-07-11, 확정): 정지일 행 112,524건의 종가는 전부 채워져 있음(NaN 0건 — 정지일에 무효화되는 건 시가·고가·저가·거래량이고 종가는 의도적으로 보존). 최근 2년 "정지 해제 첫날" 454건 중 0.2% 이상 하락 242건, 거래량 조건까지 충족 64건 — 그 64건이 전부 현재 flag=True로 잡혀 있습니다. 즉 해제일 탐지는 원래도 되고 있었고, 새 컷(−0.2%)에서도 동일하게 됩니다.

남는 조치: volume.py docstring의 "알려진 잔여 비정합: halt 직후 첫 거래일 return은 NaN → flag False" 문구가 사실과 다르므로 삭제/정정 필요(코드 동작 변경 없음, 주석만). 낡은 주석을 방치하면 나중에 이 주석을 근거로 잘못된 판단을 하는 — 정확히 이번에 겪은 류의 — 사고가 재발합니다. 이슈 #30 본문의 항목④에도 정정 코멘트를 남깁니다.

6. 제안 실행 절차 (순서 중요)

핵심 요약 ① 크론 PATH 수리(선행 — 안 하면 내일부터 또 안 돎) → ② 밀린 4거래일 데이터 따라잡기(수동 1회 실행) → ③ 전체 재계산 → ④ 전후 수치 측정·기록 → ⑤ 사실과 다른 주석 정정. 전부 로컬 작업이고 되돌릴 수 없는 단계는 없습니다.
  1. 크론 수리: 크론탭 맨 위에 PATH=/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin 1줄 추가.
  2. 따라잡기: --chain=daily·market_context 수동 1회 실행(증분 창이 30일이라 4일 공백은 한 번에 메워짐). 주말이므로 오늘 실행분으로 금요일(07-10)까지 채워짐.
  3. 재계산: 위 §2의 full-refresh 2건 실행.
  4. 기록: §3·§4 수치를 이슈 #30 코멘트 + 점검표 이력에 남김.
  5. 주석 정정: 항목④ 검토로 사실과 다르게 판명된 docstring 문구 정정(초소형 커밋).

결정해야 할 사항 — 3건 (+1건은 검토 중 해소됨)

핵심 요약 D0(크론 수리)·D1(전체 재계산)·D3(주석 정정)은 전부 "실행 승인"이고, D2(수치 기록 위치)는 관례 확인 수준입니다. 원래 설계 선택이었던 "정지 해제일 정책"은 검토 결과 전제가 틀린 것으로 판명되어 결정할 필요가 없어졌습니다(탭1 §5). 모두 되돌릴 수 있는 작업입니다.

D0. 크론 PATH 수리를 승인할까요? 선행 — 가장 급함

무엇을: 크론탭 맨 위에 PATH 한 줄을 추가해 uv를 찾을 수 있게 함. 이걸 안 하면 매일 저녁 데이터 수집이 계속 실패합니다(이미 4거래일 결손).
선택지: (a) PATH 헤더 1줄 추가 — 권장 (모든 작업에 일괄 적용, 단순) (b) 각 크론 라인의 uv를 절대경로 /opt/homebrew/bin/uv로 교체 (크론 라인마다 수정).
참고 관례: 이 리포에서 크론탭 반영은 "수동 + 운영 해설서 부록 갱신 동반"이 관례였습니다. 승인 주시면 제가 크론탭 수정과 해설서 갱신까지 함께 진행합니다.

D1. 전체 재계산(full-refresh) 실행을 승인할까요? 이슈 #30 본체

무엇을: 일봉·주봉 지표를 전 종목·전 기간 새 코드로 재계산(위 §2 명령). production DB를 덮어쓰지만 결정론 계산이라 안전하고 재실행 가능합니다. 소요는 수십 분 단위 예상.
순서 조건: D0·따라잡기(밀린 4일 적재) 후에 실행해야 "오늘까지" 정합해집니다.
안 하면: 백테스트·과거 조회가 옛 기준 flag를 영구히 보게 되고, 최근 60일 창도 약 60거래일(10월 초경)까지는 신구 기준이 섞인 채 LLM에 들어갑니다.

D2. 전후 수치(B-수치)는 어디에 기록할까요? 관례 확인

권장: 이슈 #30 코멘트에 상세 수치 + 점검표(checklist) 적용 이력에 1줄 요약 — 상세는 이슈에, 규범 문서에는 요약만 남기는 기존 관례와 일치합니다. 특별히 다른 위치를 원하시면 알려주세요.

D3. 사실과 다른 코드 주석 정정을 승인할까요? 해소된 항목의 뒷정리
