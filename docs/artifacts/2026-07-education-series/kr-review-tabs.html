1. 이 프로젝트는 무엇을 하는 시스템인가요?

kr-by-claude는 한국 주식(코스피/코스닥) 중에서 "지금 사볼 만한 종목"을 자동으로 골라내는 시스템입니다. 미국의 유명 트레이더 마크 미너비니(Minervini)와 윌리엄 오닐(O'Neil)이 책으로 정리한 종목 선정 규칙을 컴퓨터 프로그램으로 옮긴 것입니다.

전체 과정은 크게 3단계입니다.

  1. 데이터 수집: 매일 저녁, 한국거래소(KRX)에서 전 종목(~2,550개)의 주가와 거래량을 내려받아 데이터베이스에 저장합니다.
  2. 1차 선별(스크리너): 수학 공식으로 계산 가능한 조건(예: "주가가 200일 평균선 위에 있는가")을 전 종목에 적용해, 조건을 통과한 종목만 추립니다. 이 단계는 사람이나 AI의 판단 없이 숫자 계산만으로 이루어집니다. 그래서 "결정론(deterministic) 스크리너"라고 부릅니다 — 같은 입력이면 언제나 같은 결과가 나온다는 뜻입니다.
  3. 2차 판단(AI 분석): 1차를 통과한 종목의 차트 이미지를 Claude(AI)에게 보여주고, "이 차트가 매수하기 좋은 모양인가?"를 판정하게 합니다. AI는 종목마다 entry(진입 후보) / watch(관찰) / ignore(무시) 같은 등급을 매기고, 매수 기준가(pivot)도 산출합니다.

그 뒤 평일마다 "관찰 중이던 종목이 오늘 매수 기준가를 돌파했는가"를 검사하고, 돌파하면 AI에게 한 번 더 "지금 사도 되는가"를 묻고(go_now/wait/abort), 사도 된다면 매수가·손절가·목표가를 계산해 Slack 메신저로 알림을 보냅니다. 이 모든 과정은 cron(정해진 시각에 프로그램을 자동 실행해 주는 예약 장치)으로 매일 자동 실행됩니다. 단, 현재는 모의 실행(dry-run) 모드라서 실제 알림/기록 없이 연습 가동 중입니다.

데이터 흐름 (매일 저녁) KRX 주가 수집 ─→ 지표 계산 ─→ 결정론 스크리너 ─→ [주말] AI 차트 분류 18:30 (자동) (자동) (토 03:20) │ 시장 상태 계산 ─────────────────────────────┐ ▼ 19:30 ├→ [평일 20:00] 돌파 검사 │ → AI "지금 살까?" 판단 │ → 매수 파라미터 계산 └ → Slack 알림 (현재 dry-run)

2. 먼저 알아두면 좋은 용어 사전

파이프라인 (pipeline)
여러 처리 단계를 순서대로 이어 붙인 것. 공장 컨베이어벨트처럼 "수집 → 계산 → 선별 → 판단"이 차례로 흘러갑니다.
pivot (피벗, 매수 기준가)
AI가 차트를 보고 정하는 "이 가격을 넘으면 상승 추세가 확인된 것"이라는 기준 가격. 주가가 이 선을 넘는 순간을 "돌파(breakout)"라고 부르고, 그때가 매수 후보 시점이 됩니다.
go_now / wait / abort
돌파가 일어났을 때 AI가 내리는 세 가지 결정. go_now = 지금 매수해도 좋음, wait = 아직 기다림, abort = 이 종목은 포기.
raw / adj (원본 가격 / 수정 가격)
회사가 주식을 쪼개면(액면분할) 주가 숫자가 하루아침에 1/10로 보이는 왜곡이 생깁니다. 이를 보정해 과거~현재를 연속으로 이어 붙인 값이 adj(수정 주가/수정 거래량), 보정 전 원본이 raw입니다. 차트 분석에는 반드시 adj를 써야 하며, 한 화면에 둘을 섞으면 "같은 날인데 숫자가 두 개"인 모순이 생깁니다.
프롬프트 (prompt)
AI에게 주는 지시문. 이 프로젝트에서는 "차트를 이렇게 분석해라"라는 긴 지시문이 별도 텍스트 파일(prompts/*.md)로 관리됩니다. 프롬프트 안에도 기준 숫자("거래량 1.4배")가 적혀 있어서, 코드의 숫자와 어긋나지 않게 관리하는 것이 과제입니다.
payload (페이로드)
AI에게 분석을 시킬 때 함께 건네는 자료 묶음. 최근 주가 표, 계산된 지표, 시장 상태 등을 정리한 JSON(구조화된 텍스트) 형식입니다. 차트 이미지, daily.csv(표 형식 시세 파일)와 함께 한 세트로 전달됩니다.
FTD / 분배일
시장 전체 상태를 판정하는 두 가지 신호. FTD(팔로우스루 데이)는 하락장이 끝나고 상승 전환을 확인해 주는 급등일, 분배일은 반대로 기관이 물량을 파는 것으로 보이는 하락일입니다. 이 신호들이 쌓여 "상승장/하락장" 판정(market_context)이 매일 갱신됩니다.
cron / crontab
컴퓨터에 "매일 몇 시에 이 프로그램을 실행해라"라고 예약해 두는 장치(cron)와 그 예약 목록(crontab).
dry-run (모의 실행)
실제 저장·알림 없이 "돌아가는 흉내"만 내는 연습 모드. 실수 비용 없이 시스템을 점검할 때 씁니다.
백테스트 (backtest)
"이 규칙대로 과거에 투자했다면 돈을 벌었을까?"를 과거 데이터로 검증하는 모의시험.
SSOT (Single Source of Truth, 단일 진실 원천)
같은 숫자(예: "거래량이 평균의 1.4배 이상")를 여러 파일에 흩어 적으면 하나만 고치고 나머지를 빠뜨리는 사고가 납니다. 그래서 숫자를 한 파일(thresholds.py)에만 정의하고 나머지는 거기서 가져다 쓰게 하는 원칙입니다.
fail-closed / fail-open
문제가 감지됐을 때 fail-closed는 "일단 멈춤"(안전 우선), fail-open은 "경고만 하고 계속 진행"(가용성 우선). 돈이 걸린 시스템은 대부분 fail-closed가 안전합니다.
침묵 실패 (silent failure)
실패했는데 에러 메시지조차 없이 조용히 지나가는 것. 가장 위험한 종류의 버그입니다. "실패 0건"이라는 보고가 사실은 "실패를 셀 줄 모름"일 수 있기 때문입니다.
커밋 (commit) / .gitignore
커밋은 코드 변경사항을 git(버전 관리 장부)에 기록하는 것. .gitignore는 "이 파일들은 장부에 올리지 마라"는 제외 목록입니다.
테스트 격리 (test isolation)
자동 테스트끼리 서로의 흔적(데이터)을 남기지 않게 하는 것. 격리가 깨지면 "내 코드는 멀쩡한데 앞 테스트가 남긴 쓰레기 데이터 때문에 실패"하는 일이 생깁니다.
P1 / P2 / P3
우선순위 등급. P1 = 정확성에 직접 영향, 가장 먼저. P2 = 중요하지만 당장 급하지 않음. P3 = 여유 있을 때.

3. 지금 상태에 대한 총평

결론부터: 이 프로젝트는 잘 만들어져 있습니다. 원본 리뷰의 첫 문단이 말하는 "설계 성숙도가 높다"는 평가의 근거를 풀면 이렇습니다.

  • 사고에서 배운 흔적이 코드에 있다: 과거에 데이터 오염 사고를 겪을 때마다(예: 거래정지일의 0원 주가가 지표를 오염시킨 사건), 같은 사고가 재발하지 못하게 막는 안전장치를 겹겹이 추가해 왔습니다. 원본 리뷰에 나오는 어려운 단어들은 전부 이런 안전장치의 이름입니다:
    • chokepoint — 모든 데이터가 반드시 통과해야 하는 단일 검문소
    • 트립와이어 — 누군가 검문소를 우회하면 울리는 경보
    • 자가치유 — 오염 데이터를 발견하면 스스로 지우고 다시 만드는 로직
    • freshness 게이트 — 데이터가 오늘 것인지 확인하고, 낡았으면 실행을 멈추는 문
    • 멱등 재개 — 중간에 끊긴 작업을 끊긴 곳부터 다시 시작해도 결과가 같게 하는 성질
    • run 추적 — 모든 실행의 성공/실패를 남기는 기록부
  • 검증 방법론이 과학적이다: 백테스트를 할 때 "결과를 보고 나서 기준을 바꾸는" 자기기만을 막으려고, 실험 전에 판정 기준을 미리 등록하고(사전등록) 잠근 뒤 결과를 측정합니다. 실제로 기대에 어긋난 실험(v4 파일럿)을 기준대로 기각한 기록이 남아 있습니다 — 규칙을 만들기만 한 게 아니라 지켰다는 증거입니다.
  • 숫자 관리가 깨끗하다: 책에서 가져온 기준값들이 SSOT 원칙대로 한 파일에 모여 있고, 웹 화면용 사본과의 동기화도 어긋남 없이(diff 0) 유지되고 있습니다.
  • 기반도 건강하다: 웹 코드는 타입 검사(tsc — 자료형이 어긋난 코드를 잡아내는 자동 검사)를 오류 0으로 통과하고, 사용 중인 라이브러리들도 모두 최신 버전대라 알려진 보안 취약점이 없습니다.

그럼에도 남아 있는 문제들은 두 가지 공통 패턴으로 묶입니다.

패턴 ① "빈 결과가 조용히 통과한다". 외부 서버(KRX)에 데이터를 요청했는데 에러 대신 빈 응답이 돌아오는 경우가 있습니다. 지금 코드는 "에러"는 잘 잡지만 "빈 것"은 정상 취급하고 넘어갑니다. 실제로 과거에 이 문제로 607개 종목의 과거 데이터가 누락됐는데 시스템은 "실패 0건"이라고 보고한 사고가 있었고, 그 구조적 원인이 아직 코드에 남아 있습니다.

패턴 ② "코드보다 문서가 낡았다". 코드는 계속 고쳐졌는데 프로젝트 설명 문서(ROADMAP)는 5주 전에 멈춰 있고, "아직 안 고쳐진 문제" 목록에는 이미 고쳐진 것 3건이 남아 있었습니다(이번 리뷰에서 확인 후 기록을 갱신했습니다). 문서가 낡으면 다음 작업자가 이미 해결된 문제를 다시 파거나, 존재하는 기능을 모르고 새로 만드는 낭비가 생깁니다.

배경: 이 시스템은 AI(Claude)를 명령줄 도구(CLI)로 호출하는데, AI 사용에는 하루 사용량 한도가 있습니다. 한도에 도달하면 "한도 초과" 에러가 돌아옵니다. 코드에는 이를 위한 전용 에러 타입 UsageLimitError가 준비돼 있습니다.

무엇이 문제인가요?

평일 파이프라인의 세 단계 중 첫 단계(신규 종목 분석, daily_delta)는 한도 초과를 만나면 "더 해봐야 소용없으니 전체를 멈추고 실패로 기록"하도록 올바르게 짜여 있습니다. 그런데 그다음 두 단계 — "지금 살까?" 판단(evaluate)과 매수 파라미터 계산(entry) — 는 한도 초과를 일반 오류와 똑같이 취급합니다. 그래서 한도에 걸린 날에도 남은 종목들을 계속 호출하고(전부 실패), 마지막에 실행 기록은 "성공"으로 남습니다.

가장 나쁜 시나리오: 어떤 종목이 오늘 매수 적기(go_now)였는데, 한도 초과로 판단 자체가 실행되지 못했고, 시스템은 성공했다고 기록했으니 아무도 눈치채지 못합니다. 사야 할 종목을 놓쳤는데 놓친 줄도 모르는 상황입니다.

실제 코드와의 연결

올바른 처리의 견본이 이미 같은 코드베이스에 있습니다: kr_pipeline/llm_runner/daily_delta.py:63의 except UsageLimitError: raise(한도 초과면 즉시 중단하고 위로 전파). 문제 지점은 evaluate_pivot.py:127-134와 entry_params.py:95-100 — 둘 다 except Exception(모든 오류를 한 덩어리로) 하나뿐이라 한도 초과가 그 안에 삼켜집니다.

어떻게 고치나요

두 파일에 daily_delta와 똑같은 4줄(한도 초과면 되돌리고 즉시 중단)을 추가하면 됩니다. 수정량은 작지만, 실전 가동 전에 반드시 필요해서 P1 중에서도 1순위입니다.

처리 내역 (2026-07-07 완료)

계획대로 두 단계(evaluate·entry)에 1단계와 똑같은 처리를 넣었습니다. 이제 한도에 걸리면 남은 종목을 더 부르지 않고 바로 멈추고, 그날 실행을 "실패"로 기록합니다. 실패 표시가 뜨니까 사람이 알아채고 다시 돌릴 수 있고, 다시 돌리면 이미 끝낸 종목은 건너뛰고 못 한 종목만 이어서 처리합니다.

P1-2. AI가 정한 "매수 기준가"를 아무도 검산하지 않는다

배경: AI는 가끔 숫자를 틀립니다(가격 자릿수를 잘못 읽는 등). 그래서 AI가 낸 숫자는 저장 전에 "말이 되는 숫자인지" 기계적으로 검사(sanity check)해야 합니다. 예: 가격이 0보다 큰가? 손절가가 매수가보다 낮은가?

무엇이 문제인가요?

이 시스템에서 AI가 숫자를 내는 곳은 두 층입니다. 마지막 층(매수 파라미터)은 이미 검사가 잘 붙어 있습니다 — 치명적 오류(가격≤0, 손절가≥매수가 등)는 저장 자체를 거부하고, 애매한 값(책의 권장 범위 이탈)은 경고 표시를 답니다. 이번 리뷰에서 확인해 보니 이 부분은 예전 기록("검증 없음")과 달리 이미 해결돼 있었습니다.

문제는 첫 층(주말 차트 분류)입니다. AI가 여기서 정하는 pivot_price(매수 기준가), base_high/base_low(차트 패턴의 윗선/아랫선)는 아무 검사 없이 저장됩니다. 그런데 이 pivot이야말로 이후 모든 판단의 출발점입니다 — 평일마다 "주가가 pivot을 넘었나?"를 기계가 비교하고, base_low는 손절 기준으로 쓰입니다. pivot 하나가 오염되면(예: 자릿수 오류로 10배 큰 값) 그 종목은 영원히 돌파가 안 일어나거나(기준가가 터무니없이 높음) 매일 가짜 돌파가 일어납니다(터무니없이 낮음). 최근 수행한 "놓친 승자 감사"에서 병목이 바로 이 돌파 검사 경로로 지목됐는데, 이 문제는 그 경로의 입력 데이터를 오염시킬 수 있는 위치에 있습니다.

덧붙여, 마지막 층에도 남은 구멍이 하나 있습니다: AI가 보고한 현재가(current_price)를 데이터베이스에 있는 실제 종가와 비교하지는 않습니다. AI가 가격을 크게 오독해도 내부 논리만 맞으면(손절<매수<목표) 통과되어 Slack 매수 알림까지 그대로 나갈 수 있습니다.

실제 코드와의 연결

잘된 예: kr_pipeline/llm_runner/store.py:389-443의 _validate_entry_params_sanity(매수 파라미터 검사). 구멍: 같은 파일의 insert_classification(:94)은 분류 등급의 종류(enum)와 위험 플래그 이름만 확인하고 가격 숫자는 그대로 저장. 그 pivot을 소비하는 곳: trigger_gate.py:83(종가>pivot 비교), load.py:164(base_low를 손절가로 사용).

어떻게 고치나요

분류 저장 시 최소 규칙 추가: pivot>0, base_low>0, base_low<base_high, base_low≤pivot (위반 시 저장 거부). 여기에 "pivot이 최근 종가와 수십% 이상 동떨어지면 경고"를 더하면 자릿수 오류도 잡힙니다. 매수 파라미터 쪽에는 "AI의 current_price와 DB 실제 종가가 ±5~10% 이내인지" 교차검증을 추가합니다.
