?먼저, "백필(backfill)"이 뭔가요?

backfill = 빈칸 채우기

이 시스템은 원래 매주 토요일마다 자동으로 주식들을 분석합니다(주말 루틴). 그런데 가끔 과거에 분석이 안 된 날짜가 있거나, "2024년 5월 한 달치를 다시 분석해 보고 싶다" 같은 상황이 생기죠. 이때 사람이 직접 날짜를 골라 과거 분석을 메꾸는 것이 바로 "수동 백필"입니다.

🕐 핵심 비유 — 시간여행
과거의 특정 토요일로 시간여행을 가서, 그 시점까지의 데이터만 가지고 주식을 분석합니다. 지금 알고 있는 "미래 정보"는 절대 보지 않아요. 그래야 "그때 판단했다면 맞았을까?"를 공정하게 확인할 수 있으니까요. (시험 볼 때 정답지 안 보는 것과 같죠.)

이 작업은 코드에서 llm-backfill 이라는 이름(id)을 가지고 있고, UI에는 LLM 백필 (수동) 으로 표시됩니다.

▦전체 흐름 한눈에

색깔: 🟣 화면(UI) · 🔵 서버(API) · 🟢 분석 프로그램 · 🟡 데이터베이스

아래에서 각 단계를 자세히 살펴봅니다.

①버튼 클릭과 설정 창

UI

설정 창에서 3가지를 고른다

web/src/components/RunDialog.tsx

"수동 실행" 버튼을 누르면 수동 실행 — LLM 백필 (수동) 이라는 제목의 작은 설정 창이 뜹니다. 여기서 정하는 것:

1) 모드 — 둘 중 하나

  • 미리보기 (dry-run) — AI를 진짜로 부르지 않고, "이번에 몇 개 종목이 대상인지"만 확인. 비용 0원, 연습 경기. (코드 인자: --mode=backfill --dry-run)
  • 실제 분류 (real) — 진짜로 AI를 부르고 결과를 저장. 비용 발생, 본 경기. (코드 인자: --mode=backfill)

2) 시작일 / 종료일

어느 기간을 분석할지 (예: 2024-05-01 ~ 2024-05-31). 이 둘은 필수예요 — 비워두면 "실행" 버튼이 눌리지 않습니다.

3) 종목

특정 종목 코드들을 쉼표로 구분해 넣거나(예: 005930,000660), 비워두면 "조건을 통과한 전 종목"이 대상이 됩니다.

⚠️ 비용 경고는 "실제 분류"에서만 뜹니다. 종목을 비운 채 실제 분류 모드로 실행하려 하면 "전 종목 백필은 LLM 비용이 큽니다. 정말 실행하시겠습니까?" 확인창이 뜹니다. AI를 수백 번 부르면 비용이 크니까요. (미리보기 모드는 비용이 없어서 이 경고가 나오지 않습니다.)

②서버로 주문 전달

API

주문표를 주방에 넘긴다

web/src/components/RunDialog.tsx → api/routers/runner.py

"실행"을 누르면 화면은 서버에 주문서를 보냅니다 (POST /api/runner/run). 주문서 내용은 이런 모양이에요:

// 웹 → 서버로 보내는 주문서 (JSON)
{
  "pipeline_id": "llm-backfill",   // 어떤 작업?
  "mode_id": "real",             // "dry-run" 또는 "real"
  "force": false,                // 강제 재실행?
  "params": {                      // 사용자가 채운 값들
    "start": "2024-05-01",
    "end": "2024-05-31",
    "tickers": ""             // 비어있음 = 전 종목
  }
}
🍽️ 식당 비유 — 손님(화면)이 주문표를 작성해 주방(서버)에 넘긴 단계입니다.

③중복 주문 막기 (안전장치)

API

똑같은 요리를 두 번 만들지 않기

api/services/runner_service.py → check_can_run_pipeline()

주방은 무작정 요리를 시작하지 않습니다. 먼저 두 가지를 확인해요:

  1. 지금 같은 작업이 이미 돌고 있나? → 그러면 거절 (409 에러: "이미 실행 중입니다")
  2. 이미 성공적으로 끝낸 적 있나? → 그러면 거절 ("같은 데이터 날짜의 작업이 이미 성공했습니다")

단, 설정 창의 "force(강제 재실행)" 체크박스를 켜면 2번 검사를 건너뛰고 무조건 실행합니다.

"이미 끝났나?"의 정확한 기준은 날짜입니다. 사람이 느끼는 "오늘"이 아니라, 시스템이 가진 최신 데이터 날짜(as_of) 를 기준으로 같은 작업이 성공한 적 있는지 봐요. (pipeline_runs 테이블의 기록을 조회.)
🍳 비유 — 같은 요리를 두 번 만들어 재료(LLM 비용)를 낭비하지 않으려는 장치입니다.

④백그라운드에서 프로그램 실행

PROC

불 붙여놓고 신경 끄기

api/services/runner_service.py → spawn_pipeline()

검사를 통과하면, 서버는 실제 분석 프로그램을 백그라운드로 따로 띄웁니다. 실제로 만들어져 실행되는 명령어는 이렇게 생겼어요:

# 서버가 자동으로 조립해 실행하는 명령어
uv run python -m kr_pipeline.llm_runner \
  --mode=backfill \
  --start=2024-05-01 \
  --end=2024-05-31
  # tickers 가 비어있으면 --tickers 인자는 안 붙음

중요한 점: 이것은 "fire-and-forget(불 붙이고 잊기)" 방식이에요. 서버는 "작업 번호(PID) 줄게, 진행 상황은 따로 확인해" 하고 손님에게 곧바로 답을 줍니다. 요리는 주방에서 알아서 진행되고, 손님은 화면에서 진행률을 새로고침하며 봅니다.

왜 이렇게 하나요? 전 종목 백필은 몇 분~몇 시간 걸릴 수 있어요. 손님을 그동안 문 앞에 세워둘 순 없으니, 주문만 받고 "번호표" 주고 돌려보내는 거죠.

⑤토요일마다 반복 (핵심 루프)

PROC

(a) 기간 안의 모든 토요일을 뽑는다

kr_pipeline/llm_runner/backfill.py → _enumerate_saturdays()

예: 2024-05-01 ~ 05-31 → [5월 4일, 11일, 18일, 25일] (토요일 4개)

왜 토요일이냐면, 이 투자 전략(Minervini / O'Neil)은 주말에 차분히 한 주를 복기하는 루틴이기 때문입니다.

PROC

(b) 그날 "분석할 가치가 있는 종목"만 추린다

kr_pipeline/llm_runner/load.py → get_qualifying_tickers()

아무 종목이나 분석하지 않아요. Minervini 추세 템플릿 조건을 통과한 종목만 골라냅니다. 예를 들면:

  • 주가가 150일 평균선 위에 있고
  • 장·단기 평균선들이 우상향이고
  • 상대강도(다른 종목 대비 강함)가 충분히 높고… 등등
중요한 세부 — 토요일엔 장이 안 열린다! 그래서 "그 토요일 날짜"의 지표를 그대로 쓰는 게 아니라, 그 날짜 이하 가장 최근 거래일(보통 직전 금요일)의 지표를 찾아서 씁니다. (SELECT MAX(date) ... WHERE date <= 토요일) 즉 "그 주가 끝난 직후 시점"으로 분석하는 셈이에요.
🩺 비유 — "건강검진 1차를 통과한 사람만 정밀검사로 보낸다." 기계적(결정론) 1차 필터로 거른 뒤, AI라는 비싼 정밀검사를 돌리는 거예요.
PROC

(c) 이미 분석한 종목은 건너뛴다

kr_pipeline/llm_runner/backfill.py → _already_backfilled()

그 토요일에 대해 이미 분석 결과가 DB에 있으면 다시 하지 않습니다. 그래서 중간에 멈췄다 다시 돌려도 처음부터가 아니라 이어서 진행돼요. (시간여행 복습을 두 번 안 함 → "멱등성".)

⑥종목별 "분석 자료 ZIP" 포장

PROC

AI에게 보낼 자료 묶음을 만든다

api/services/zip_builder.py → build_analysis_zip()

종목 하나하나에 대해, AI에게 보낼 자료 묶음(ZIP 파일)을 만듭니다. 백필에서는 14개 파일이 들어가요(과거 분석 결과는 빼고 — 시간여행이니까):

README.md
사용 설명서
prompt_step1_analyze.md
분석 지시문(룰)
prompt_step2_entry_params.md
진입 파라미터 지시문
prompt_verify.md
검증용 지시문
payload.json
핵심 지표 통합본
market_context.json
시장 상황(상승/하락장)
corporate_actions.json
액면분할·합병 이력(5년)
minervini.json
8조건 통과 상세
daily.csv
최근 60일 가격
weekly.csv
최근 104주(2년) 가격
market_index_daily.csv
시장지수 일봉
market_index_weekly.csv
시장지수 주봉
daily_chart.png
📈 1년 일봉 차트(그림)
weekly_chart.png
📈 2년 주봉 차트(그림)
가장 중요한 포인트 — 미래를 보지 않는다. 이 자료들은 전부 "그 토요일 시점" (on_date)으로만 만들어집니다. 14개 파일의 모든 데이터 쿼리에 WHERE date <= on_date 상한이 걸려, 차트도 그 시점까지만 그리고 그 이후의 미래 데이터는 한 톨도 안 들어가요. 그래야 백테스트가 공정합니다.
⚠️ 한 파일은 상한이 빠져 있었다 (수정됨). 다른 13개 파일과 달리 corporate_actions.json(액면분할·합병 이력)은 한동안 하한선만 있고 상한선(event_date <= as_of)이 없었습니다. 그래서 과거 백필 시 분석 시점 이후에 일어난 분할이 ZIP에 섞여 들어가 "최근 12주 내 분할" 플래그까지 잘못 켜질 수 있었어요(look-ahead bias). 이 문서를 검증하는 과정에서 발견해 corporate_actions_builder.py 에 상한선을 추가하고 회귀 테스트로 막았습니다. 지금은 14개 파일 전부 미래를 보지 않습니다.
숨은 안전장치 — 데이터 무결성 검사. ZIP을 만들기 직전에 check_data_integrity 가 돌아, 그 종목의 데이터가 손상/누락됐는지 점검합니다. 이상이 있으면 그 종목은 (다음 단계 ⑨의 fail-soft로) 건너뜁니다.

⑦Claude AI에게 분석 요청

PROC

포장한 ZIP을 AI에게 보낸다

kr_pipeline/llm_runner/llm/claude_cli.py → call_claude()
  • 미리보기(dry-run) 모드면: AI를 진짜로 안 부르고 가짜 결과(mock)를 만들어 돌려줍니다. "대상이 몇 개인지"만 확인하는 연습용.
  • 실제(real) 모드면: 컴퓨터에 설치된 claude 명령어를 실제로 실행해서, 지시문(analyze_chart_v3.md)과 ZIP을 함께 보냅니다.

AI는 차트를 "눈으로 보고" 이런 판단을 JSON으로 돌려줍니다:

{
  "classification": "entry",        // entry / watch / ignore 중 하나
  "pattern": "cup_with_handle",    // 차트 패턴
  "confidence": 0.82,             // 확신도 0.0~1.0
  "pivot_price": 71500,           // 매수 기준 가격
  "reasoning": "...",             // 왜 이렇게 판단했는지
  "risk_flags": []                // 위험 요소들
}
판정의미
entry지금 사도 좋은 자리
watch통과는 했지만 아직 살 때는 아님 (관찰)
ignore무시 (이미 너무 올랐거나 모양이 나쁨)
🔁 재시도 장치. AI 호출이 실패하면 0초 → 1초 → 3초 → 9초 간격으로 최대 4번 다시 시도합니다(각 호출 제한시간 600초). 인터넷이 잠깐 끊겨도 버티도록 한 거예요. 4번 다 실패하면 그 종목만 에러로 처리됩니다.

⑧결과를 DB에 저장

DB

한 줄의 기록으로 남긴다

kr_pipeline/llm_runner/store.py → insert_backfill_classification()

AI의 답을 받으면 classification_backfill 이라는 DB 표에 한 줄로 저장합니다. 저장에는 종목코드·분석기준일(토요일)·판정·패턴·확신도·근거· AI 호출 소요시간 등이 모두 기록돼요.

미리보기(dry-run) 모드면 이 저장 단계를 통째로 건너뜁니다. 그래서 비용도 안 들고 DB도 안 더럽혀요.

저장 전 두 가지 처리

  1. Phase 1 게이트 — 저장 직전 추가 검증(위험 플래그 정리 등)을 거칩니다. 만약 이 게이트 처리가 실패해도, 원본 결과를 그대로 저장하고 넘어갑니다(fail-soft).
  2. 중복 방지 잠금 — 같은 (종목, 토요일) 조합이 두 번 저장되지 않도록 ON CONFLICT ... DO NOTHING 이 걸려 있어요.
멱등성 = 2중 안전장치. ⑤-(c)에서 이미 한 건 후보에서 빼고, 여기 ⑧에서 DB 차원에서도 중복을 막아요. 두 겹 덕분에 몇 번을 다시 돌려도 데이터가 중복되거나 망가지지 않습니다.

⑨실패해도 멈추지 않기 (Fail-soft)

PROC

한 종목 실패가 전체를 멈추지 않는다

kr_pipeline/llm_runner/backfill.py → run()

종목 하나를 분석하다 에러가 나도 전체가 멈추지 않습니다. 그 종목만 "실패 목록"에 기록하고, 다음 종목으로 넘어가요. 성공한 것들은 각각 따로 저장(commit)되고, 실패한 것은 되돌립니다(rollback).

🍽️ 비유 — 한 테이블 주문이 잘못돼도 다른 테이블 음식은 계속 나가는 식당.

모든 토요일·모든 종목이 끝나면, 프로그램은 요약을 돌려줍니다:

# 최종 요약 (예시)
{
  "weeks": 4,              // 4번의 토요일 처리
  "processed": 420,        // 420건 분석 성공
  "skipped_existing": 0,   // 0건 건너뜀(이미 있던 것)

📖용어 사전

백필 (backfill)
과거의 빈 데이터를 소급해서 채워 넣는 것. 여기선 과거 토요일들의 LLM 분류를 만드는 일.
dry-run (미리보기)
실제로 비싼 작업(AI 호출·DB 저장)을 하지 않고, 대상·흐름만 점검하는 연습 실행.
as_of (기준일)
"이 시점 기준으로" 분석한다는 날짜. 백필에서는 각 토요일이 곧 분석 기준일이 됩니다.
멱등성 (idempotency)
같은 작업을 여러 번 해도 결과가 한 번 한 것과 똑같은 성질. 다시 돌려도 안전.
fire-and-forget
"불 붙이고 잊기" — 오래 걸리는 작업을 백그라운드에 던져두고 즉시 응답을 돌려주는 방식.
fail-soft
일부가 실패해도 전체를 멈추지 않고, 실패는 따로 기록하며 나머지를 계속 진행하는 설계.
Minervini Trend Template
마크 미너비니의 추세 추종 종목 선정 기준(8가지 조건). 1차 기계 필터로 사용됩니다.
classification (분류)
AI가 내리는 판정: entry(매수 적기) / watch(관찰) / ignore(무시).
pivot (피벗) 가격
차트 패턴에서 "이 가격을 돌파하면 매수"로 보는 기준선.
