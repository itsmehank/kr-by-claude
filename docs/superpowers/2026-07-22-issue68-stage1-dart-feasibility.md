## 3. 2단계 설계에 반영해야 할 실측 발견 3건

- **A. rcept_no 가 원공시 시점을 보증하지 않는다 (look-ahead 함정)**:
  fnlttSinglAcnt 의 rcept_no 는 *최신* 접수 — 실측 예: 000760 의 FY2019~2022
  보고서 4건이 전부 rcept 2024-03(최대 +1,535일). 정정공시 덮어쓰기로 **추정**
  되나(일괄 지연 제출 가능성도 잔존 — JSON 만으로 구분 불가, 검증 리뷰 지적)
  어느 쪽이든 설계 결론은 동일: **as-of 공시일은 이 API 로 잡으면 안 되고**
  공시검색(list.json)의 *원공시* 접수일을 별도 확보해야 함. 이슈 본문의
  "공시 지연 — look-ahead 방지 기준"의 핵심 스펙.
- **B. 비12월 결산 법인 존재**: 050860 은 연간 rcept 가 8년 중 7년 9월(1회 12월,
  전부 음수 lag) — 비12월 결산. 응답의 `thstrm_dt`(예: "2023.01.01 ~
  2023.12.31")를 파싱하면 실제 회계기간 확보 가능(세션 중 1건 실사 — 프로브
  JSON 미수록, 2단계 적재 시 저장 필드로 승격할 것) — 회계기간 하드코딩 금지.
- **C. 계정명 변형 + EPS 부재**: 순이익 계정명은 "당기순이익(손실)" — 직매칭
  "당기순이익"은 성공 응답 273건 중 **0건** 매칭(프로브 집계 실측). 주요계정
  API 에 **EPS 없음** — C 기준(분기 EPS ≥ +25%)은 ①파생(순이익 ÷ 주식수,
  stockTotqySttus API) 또는 ②전체계정(fnlttSinglAcntAll)에서 주당이익 추출 중
  택일 — 2단계 결정 항목. 당기순이익의 지배주주 구분 중복 행도 세션 중 1건
  실사에서 관찰(JSON 미수록) — 정규화 규칙 필요.
